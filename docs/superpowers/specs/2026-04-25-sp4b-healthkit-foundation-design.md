# SP4b — HealthKit Foundation Design

**Date:** 2026-04-25
**Status:** Draft, pending user review
**Parent:** [`./meta/2026-04-25-ios-v2-workouts-design.md`](./meta/2026-04-25-ios-v2-workouts-design.md) (SP4 meta-spec, slice 4b)
**Scope:** Native dev-client toolchain proven on user's iPhone, plus a thin HealthKit wrapper module that 4d/4e will consume. No feature UI.

---

## 1. What 4b ships

- **Local-Mac toolchain proven**: `expo prebuild` → Xcode build → sideload to user's iPhone with personal team (free Apple ID). 7-day re-sign cycle accepted.
- **`@kingstinct/react-native-healthkit` v9** (Nitro-based, new-arch ready) wired in via its Expo config plugin — Info.plist usage strings and `Health.entitlements` are auto-generated.
- **Thin wrapper module** at `lib/health/*` exposing the narrow API that 4d/4e need: `requestPermissions()`, `writeWorkout(payload)`, `useLiveHeartRate()`.
- **Dev-only screen** at `app/dev/healthkit.tsx`: buttons to request permission, write a hand-rolled `HKWorkout`, and a live HR readout.
- **Smoke test**: dev client launches on iPhone → screen renders → Apple's HealthKit permission sheet appears → "Write workout" inserts a 5-min row visible in Health.app → "Start HR" displays a live BPM number while wearing Apple Watch.

**Smoke success criterion**: every step above passes on a clean install. If permission is silently denied, the dev screen surfaces the auth error explicitly.

## 2. Locked decisions

| Decision | Choice | Reason |
|---|---|---|
| Build path | Local Mac + Xcode + personal team signing (free Apple ID) | User has Mac; avoids paid account and EAS Build. 7-day re-sign accepted for personal app. |
| HealthKit library | `@kingstinct/react-native-healthkit` v9 | Rebuilt on Nitro Modules, new-arch native, actively maintained (Feb 2026 releases include Xcode 26 fixes), Expo config plugin handles Info.plist + entitlements. |
| `react-native-health` rejected | Not new-arch first-class; project already has `newArchEnabled: true`. |
| Smoke surface | Dev-only screen at `app/dev/healthkit.tsx` | Explicit, isolated from production routes, easy to delete after SP4. |
| Smoke write payload | `HKWorkout` only (type, start, end, duration) | Smallest viable HealthKit write. `activeEnergyBurned` / `distance` are 4e's concern. |
| Smoke read approach | Anchored-object query polling ~5 s for `HKQuantityTypeIdentifierHeartRate` | Proves HR streams in. Real session-HR strategy is 4d's concern. Avoids background-delivery entitlements that may need paid account. |
| `ios/` checked into git | **No, gitignored** | Re-prebuild after dependency changes. Keeps repo small, follows managed-Expo convention. Trade-off: signing must be re-set in Xcode after each prebuild. |
| Permission grant model | `requestPermissions()` returns a single `granted` boolean | Apple deliberately hides per-type grants. Real handling is "try the op, catch not-authorized" — 4d/4e learn this; smoke surfaces the error explicitly. |

## 4. Toolchain steps (preview; the plan expands these into TDD-where-applicable tasks)

1. Install deps: `@kingstinct/react-native-healthkit`, `react-native-nitro-modules`.
2. Edit `app.json`: add the kingstinct config plugin entry with `NSHealthShareUsageDescription` and `NSHealthUpdateUsageDescription` strings; enable HealthKit entitlement.
3. Run `expo prebuild --platform ios --clean` — verify generated `ios/Pulse/Pulse.entitlements` includes the HealthKit key.
4. Open `ios/Pulse.xcworkspace`, set personal team in Signing & Capabilities, confirm bundle ID `com.kael.pulse`, ensure HealthKit capability is present.
5. Build to physical iPhone via Xcode. Confirm app launches and existing Today screen renders unchanged (no regression from SP3a/SP4a).
6. Implement `lib/health/*` wrapper modules.
7. Build the dev smoke screen.
8. Run smoke verification (the five steps in §1). Commit. Update SP4 meta-spec status row.

## 5. Wrapper API surface

```ts
// lib/health/types.ts
export type WorkoutWritePayload = {
  activityType: 'traditionalStrengthTraining' | 'running' | 'rowing' | 'other';
  start: Date;
  end: Date;
  // 4e will extend: totalEnergyBurned?, totalDistance?, metadata?
};

export type HRSample = { bpm: number; sampledAt: Date };

// lib/health/permissions.ts
export async function requestPermissions(): Promise<{ granted: boolean }>;

// lib/health/workouts.ts
export async function writeWorkout(p: WorkoutWritePayload): Promise<void>;

// lib/health/heart-rate.ts — React hook
export function useLiveHeartRate(opts?: { pollMs?: number }): {
  current: HRSample | null;
  isStreaming: boolean;
  start: () => void;
  stop: () => void;
};
```

**Why these shapes:**
- `requestPermissions` returns a single boolean because iOS deliberately hides per-type grants for privacy. Callers handle denial reactively via thrown errors, not by inspecting a per-type result.
- `useLiveHeartRate` is a hook (not a callback API) because both the dev screen and SP4d's Active Session render it. Cleanup is wired through `useEffect` return.
- The activity-type union is intentionally narrow — these are the four kinds of workouts the SP4 catalog can produce. New kinds are added when the catalog adds them, not preemptively.

## 6. Risks & fallbacks

| Risk | Likelihood | Fallback |
|---|---|---|
| Personal team rejects HealthKit entitlement at sign time | Medium | Pivot to paid Apple Developer Program ($99/yr). SP4b re-scopes to add EAS setup; the wrapper API and dev screen survive unchanged. |
| `react-native-nitro-modules` peer-dep conflicts with Expo SDK 54 | Low | Library targets recent Expo; if it breaks, pin a working version or defer SP4b and do SP4c first (web-verifiable, no native deps). |
| `expo prebuild` overwrites manual Xcode signing settings | Certain | Expected. Signing is configured per-clone, not committed. The plan documents "re-set team in Xcode after each prebuild" as a known annoyance. |
| 7-day personal-team re-sign cycle bites mid-SP4 | Certain | Accepted. User rebuilds via Xcode each week. SP4 plans must not assume long-running on-device sessions. |
| Health.app doesn't show our test workout | Medium | Most common cause: silent permission denial. The dev screen MUST surface auth errors explicitly so the smoke test can tell "wrote nothing" from "permission denied". |

**Rollback criterion**: if after one focused session the local build doesn't produce a working dev client on iPhone, stop and reassess — paid account, different toolchain, or descope SP4b and re-sequence to SP4c.

## 7. Verification posture

Per SP4 meta-spec §5, each child slice has its own smoke test. SP4b's smoke is fully on the user's iPhone (web target is meaningless for HealthKit). The smoke is described in §1 above and must pass before SP4b closes. SP4 meta-spec status row is updated as the final task of the implementation plan.

## 8. What this spec is NOT

- Not the design for any consumer screen — 4c (Routine browse + edit) and 4d (Active Session) own their UI; this spec only ships the dev smoke screen.
- Not the final shape of the workout payload — 4e adds energy/distance/metadata to `WorkoutWritePayload`.
- Not the final HR strategy — 4d picks the real polling/observer/session strategy.
- Not an EAS Build setup — deferred until/unless we adopt a paid Apple Developer membership.
- Not an implementation plan. Next step is invoking `superpowers:writing-plans` to produce the SP4b plan.
